Chapter 1 — Foundation

What is the Metasploit Framework?

The Metasploit Framework (MSF) is the world's most widely used open-source penetration testing platform. It provides a structured environment for developing, testing, and executing exploit code against remote target systems. Used by security professionals, red teams, and researchers to identify vulnerabilities before attackers do.

Chapter 2 — Internals

Metasploit Architecture — How It's Built

Metasploit is built in Ruby using a clean layered architecture. Understanding the internals helps you write custom modules, understand why things work, and troubleshoot complex scenarios.

MSF ARCHITECTURE DIAGRAM — FULL STACK

The Module System

Everything in Metasploit is a module — a self-contained Ruby class with metadata (name, author, CVE, platform, rank) and a run() or exploit() method. Modules live in ~/.msf4/modules/ (user) or /usr/share/metasploit-framework/modules/ (system).

Rex Library

Rex (Ruby Extension Library) provides low-level building blocks: Rex::Socket for networking, Rex::Proto::* for 40+ protocol implementations (SMB, HTTP, DNS, FTP, SSH...), Rex::Crypto for encryption. All modules use Rex internally.

PostgreSQL Database

MSF stores all pentest data in PostgreSQL: discovered hosts, open ports, service versions, vulnerability results, captured credentials (hashes, plaintext), loot files, and active sessions. Use workspace to separate engagements. Query with hosts, services, vulns, creds.

Chapter 5 — Payloads

Payloads & Shellcode — What Runs on the Target

A payload is the code that runs on the target after a successful exploit. It can open a shell, run a command, download a file, or provide full Meterpreter control. Understanding payload types and msfvenom is critical for real-world penetration testing.

Auxiliary Modules & Post-Exploitation

Auxiliary modules perform tasks without needing an exploit — scanning, brute-forcing, sniffing, and fuzzing. Post modules run after a successful exploit to gather information, escalate privileges, and establish persistence.

Chapter 9 — Evasion

Modern defences — antivirus (AV), endpoint detection & response (EDR), and intrusion detection systems (IDS) — detect known payloads by signature, behaviour, or heuristics. Evasion techniques are essential for realistic red team assessments against hardened environments.

Why Signatures Fail

AV products maintain signature databases of known malicious byte patterns. When Metasploit payloads are encoded with shikata_ga_nai, each iteration produces a different byte sequence — defeating static signature matching. The polymorphic nature means each generation looks different.

Behavioural Detection

Modern EDRs (CrowdStrike, SentinelOne, Carbon Black) use behavioural analysis — they observe what code does, not what it looks like. Spawning cmd.exe, injecting into lsass, calling WinAPIs for keylogging — these behaviours trigger alerts regardless of encoding. Evasion becomes much harder.

Responsible Testing Note

Testing AV bypass techniques must only be done in authorised lab environments. Never upload real payloads to VirusTotal (they are shared with AV vendors). Use isolated VMs. Document all evasion techniques in your pentest report so the blue team can improve detections.
